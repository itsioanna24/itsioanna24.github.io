Individual Presentation

Designing an Intelligent Multi-Agent System for Academic Research

Overview

This individual project required implementing the multi-agent system proposed by Group E and delivering a ten-slide presentation on the functionality and code. The system was built from scratch in Python, fully aligned with the team’s design proposal: a blackboard-coordinated, Belief–Desire–Intention (BDI) multi-agent architecture capable of autonomously retrieving academic data from the ArXiv API, evaluating relevance, and generating structured JSON outputs.

The presentation was structured to walk through the system across ten slides, covering the motivation for automating academic research tasks, the BDI cycle and blackboard coordination, the four specialised agents, the adaptive scoring and filtering logic, a live runtime walkthrough, the unit testing strategy, and a critical evaluation of strengths, limitations, and future directions.

System Implementation Summary

The full codebase was written from scratch and comprises the following components:

  • Core modules: the abstract BDI base agent (bdi_agent.py), the shared blackboard environment (environment.py), the relevance scoring function (scoring.py), and shared utilities (utils.py).
  • Specialised agents: search_agent.py, extraction_agent.py, processing_agent.py, and storage_agent.py — each implemented as a separate module reflecting the functional decomposition of the workflow.
  • Unit tests: 24 tests across 7 categories, covering all major system components and validating behaviour in isolation before full pipeline integration.

Critical Reflection

The most significant learning from this project was the difference between understanding a theoretical model and being able to operationalise it. Writing the perceive–deliberate–act cycle as a shared run method across the four agents, and using the blackboard as the sole communication channel, transformed my understanding of the BDI model from descriptive to operational. The abstractions I had read about in Rao and Georgeff (1995) and Wooldridge (2009) became concrete design decisions with real consequences for how the system behaved.

The most difficult part was compressing the full functionality into ten slides, which forced me to decide what was essential and what could be omitted without losing the design logic. This discipline proved useful: the constraint made me identify the adaptive filtering loop in the ProcessingAgent as the system’s core decision-making contribution, and I structured the presentation around demonstrating that behaviour clearly through both the slide diagrams and the runtime logs.

Designing and running the 24 unit tests also reinforced the importance of systematic validation. They surfaced edge cases I had not anticipated — particularly around the adaptive threshold behaviour when no records meet the initial criteria — built confidence that each agent behaved correctly in isolation and in combination, and gave me a repeatable way to verify the system after any change. This directly reflected software engineering practice (Sommerville, 2016) applied to an agent-based context.

The principal limitation I identified in the evaluation — that keyword-based scoring adjusts the quality bar rather than improving genuine relevance — is one I would address first in any future iteration, likely through Scikit-learn text classifiers or semantic similarity embeddings. The constraint to a single data source (ArXiv) and the absence of learning-based behaviour reflect a deliberate staged approach rather than an oversight, but they define a clear roadmap for extending the system toward a genuinely intelligent agent-based solution.
