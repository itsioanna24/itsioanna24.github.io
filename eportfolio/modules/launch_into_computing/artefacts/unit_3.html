Comparison of Insertion Sort and Merge Sort

This assignment evaluates the performance of two sorting algorithms — Insertion Sort and Merge Sort — across different input types and dataset sizes, implemented in Python. Both algorithms were tested on datasets of sizes 10, 100, 500, and 1,000 elements, across four input categories: random, sorted, reversed, and nearly sorted arrays.

Algorithm Overviews

Insertion Sort

  • Simple comparison-based algorithm.
  • Builds a sorted array one element at a time.
  • Best case: O(n) — input already sorted.
  • Average / Worst case: O(n²) — nested loop structure causes quadratic degradation.
  • Efficient for small-scale or frequently updated datasets (Qowy et al., 2022).
  • In-place; no additional memory required.

Merge Sort

  • Recursive, divide-and-conquer algorithm.
  • Splits the array in half, sorts each half, then merges.
  • All cases: O(n log n) — consistent across all input types.
  • Higher memory overhead due to auxiliary arrays during merging.
  • Applied in large-scale external sorting of ASCII records on SSDs (Kristo and Kraska, 2023).
  • Scales effectively regardless of initial element order.

Python Implementation

import time
                            import random
                            import matplotlib.pyplot as plt
                            
                            # ── Insertion Sort ──────────────────────────────────────────
                            def insertion_sort(arr):
                                for i in range(1, len(arr)):
                                    key = arr[i]
                                    j = i - 1
                                    while j >= 0 and key < arr[j]:
                                        arr[j + 1] = arr[j]
                                        j -= 1
                                    arr[j + 1] = key
                            
                            # ── Merge Sort ───────────────────────────────────────────────
                            def merge_sort(arr):
                                if len(arr) <= 1:
                                    return arr
                                mid   = len(arr) // 2
                                left  = merge_sort(arr[:mid])
                                right = merge_sort(arr[mid:])
                                return merge(left, right)
                            
                            def merge(left, right):
                                merged = []
                                i = j = 0
                                while i < len(left) and j < len(right):
                                    if left[i] < right[j]:
                                        merged.append(left[i]);  i += 1
                                    else:
                                        merged.append(right[j]); j += 1
                                merged.extend(left[i:])
                                merged.extend(right[j:])
                                return merged
                            
                            # ── Input generators ─────────────────────────────────────────
                            def generate_random(n):      return [random.randint(0, 10000) for _ in range(n)]
                            def generate_sorted(n):      return list(range(n))
                            def generate_reversed(n):    return list(range(n, 0, -1))
                            def generate_nearly_sorted(n, swaps=10):
                                arr = list(range(n))
                                for _ in range(swaps):
                                    i, j = random.sample(range(n), 2)
                                    arr[i], arr[j] = arr[j], arr[i]
                                return arr
                            
                            # ── Timing harness ───────────────────────────────────────────
                            sizes       = [10, 100, 500, 1000]
                            input_types = {"Random": generate_random, "Sorted": generate_sorted,
                                        "Reversed": generate_reversed, "Nearly Sorted": generate_nearly_sorted}
                            algorithms  = {"Insertion Sort": insertion_sort, "Merge Sort": merge_sort}
                            
                            results = {alg: {itype: [] for itype in input_types} for alg in algorithms}
                            
                            for itype, gen_func in input_types.items():
                                for size in sizes:
                                    data = gen_func(size)
                                    for alg_name, alg_func in algorithms.items():
                                        arr_copy = data.copy()
                                        start    = time.time()
                                        alg_func(arr_copy)
                                        results[alg_name][itype].append(time.time() - start)

Performance Chart

Figure 1 — Sorting algorithm runtime (seconds) vs input size (n) across four input types. Insertion Sort on reversed input (green) diverges sharply from n ≈ 200 onwards. All Merge Sort variants remain tightly grouped near zero throughout.

The chart makes the theoretical complexity difference visible. All Merge Sort lines remain flat and tightly clustered regardless of input type, confirming its O(n log n) consistency. Insertion Sort's sorted and nearly sorted lines also remain near zero — its O(n) best case performing well. However, the reversed-input line diverges dramatically from n ≈ 200 onwards, reaching ~0.021 s at n = 1,000, while the random-input line also curves upward, demonstrating the O(n²) quadratic degradation predicted by theory. The crossover point — where Merge Sort becomes faster than Insertion Sort even on random inputs — occurs around n = 200, which is consistent with the academic literature on algorithm selection thresholds.

Conclusions

The empirical observations align closely with theoretical complexity models. Insertion Sort is well-suited to small or nearly ordered inputs where its O(n) best case and low overhead make it competitive. Merge Sort, however, is the more robust choice for large or unordered datasets: its O(n log n) guarantee holds across all input types regardless of initial element order, at the cost of additional memory.

These findings are consistent with established literature. Chauhan and Duggal (2020) confirm Insertion Sort's quadratic degradation in unsorted scenarios, while Alhajri et al. (2022) demonstrate Merge Sort's superior scaling on large datasets. Real-world applications reinforce both findings: Qowy et al. (2022) applied Insertion Sort successfully in an online marketplace to sort product prices (a small, frequently updated dataset), while Kristo and Kraska (2023) demonstrated a Merge Sort variant achieving significant performance gains in large-scale external sorting on SSDs.

What was the task?

To implement and empirically compare two sorting algorithms — Insertion Sort and Merge Sort — in Python, testing them across four input types and four dataset sizes, and to analyse the results against their known theoretical complexities.

What did I learn from implementing the algorithms?

Implementing both algorithms from scratch reinforced the difference between understanding pseudocode and understanding execution. Writing the insertion_sort() in-place loop made the O(n²) behaviour intuitive — each element in the unsorted portion can shift past every element in the sorted portion, so the total work scales with the square of n. Implementing merge_sort() recursively made the divide-and-conquer structure concrete: each level of recursion halves the problem, and there are log n levels, giving O(n log n) total work across all levels. Seeing these two behaviours diverge in the chart — especially the reversed-input Insertion Sort line curving sharply away from all Merge Sort lines — made the theoretical argument viscerally clear.

What surprised you?

The crossover point was more informative than the extremes. At n = 10 or 100, both algorithms are effectively instantaneous and the practical difference is negligible. The chart shows that the meaningful divergence begins around n ≈ 200, which is precisely where real algorithm selection decisions become consequential. This reinforced that Big-O notation describes asymptotic behaviour — not absolute performance — and that real-world algorithm selection requires empirical testing alongside theoretical analysis.

How will this help in future work?

The timing harness pattern used here — generating multiple input types, looping over sizes, recording durations, and plotting — is directly reusable for benchmarking any algorithm or function. More broadly, the habit of choosing algorithms based on data characteristics rather than defaulting to one method applies widely: in data preprocessing pipelines, model training routines, and database operations, understanding the relationship between input structure and algorithmic cost is essential for writing efficient, scalable code.

Learning Outcome

Overall Reflection

This assignment demonstrated that algorithmic efficiency is not an abstract property but a measurable, input-dependent characteristic. The same algorithm — Insertion Sort — performs at O(n) on sorted data and O(n²) on reversed data: two orders of magnitude different at n = 1,000. This variability is invisible when reading pseudocode and only becomes apparent through empirical measurement.

Merge Sort's stability across all input types is what makes it the default choice in production systems handling unknown or adversarial inputs. The memory cost (O(n) auxiliary space) is the trade-off — and in memory-constrained environments, Insertion Sort's in-place property may justify its use on small or pre-sorted datasets. Understanding these trade-offs — not just which algorithm is "faster" in the abstract, but under what conditions and at what cost — is the core skill this assignment developed.

The Python implementation also reinforced a broader software engineering principle: measuring before optimising. The timing harness made it possible to confirm or refute theoretical predictions with real data, which is the correct approach whenever performance decisions matter. Assumptions based on Big-O alone, without empirical validation, can be misleading — particularly at the scale where two approaches first diverge meaningfully.
